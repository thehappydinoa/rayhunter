# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

**This is a living document.** When you discover a non-obvious, verified fact the hard way — a build failure with an opaque cause, an ordering requirement, a platform quirk, a CI behavior — add it to the Gotchas section below (or the relevant section) in the same session. Keep entries short: symptom → cause → fix. Remove entries that no longer apply.

## Project overview

Rayhunter is an EFF project for detecting IMSI catchers (cell-site simulators/stingrays). It runs on cheap mobile hotspots (Orbic RC400L, TP-Link M7350, etc.), capturing Qualcomm DIAG traffic and analyzing it with heuristics for suspicious cellular network behavior.

## Workspace layout

Cargo workspace with these crates:

- `lib/` — core `rayhunter` library: DIAG protocol parsing (`diag`), QMDL file reading (`qmdl`), HDLC framing (`hdlc`), GSMTAP conversion (`gsmtap`), pcap writing (`pcap`), and the analysis framework (`analysis/`). `diag_device` (the `/dev/diag` interface) is unix-only via `#[cfg(target_family = "unix")]` so the rest of the lib compiles on Windows.
- `daemon/` — `rayhunter-daemon`, the binary that runs **on the device**: axum web server (`server.rs`, routes in `main.rs`), recording store (`qmdl_store.rs`), analysis worker (`analysis.rs`), plus per-device modules for `display/` and `battery/`. Bundles the built frontend from `daemon/web/build` via `include_dir` — **the frontend must be built before the daemon compiles**.
- `daemon/web/` — SvelteKit + TypeScript + Tailwind frontend.
- `check/` — `rayhunter-check`, host-side CLI for offline analysis of QMDL/pcap captures.
- `installer/` — host-side CLI that installs the daemon onto devices; one module per device (`orbic.rs`, `tplink.rs`, ...).
- `installer-gui/` — Tauri GUI installer; experimental, **not** in `default-members`, so plain `cargo check/test` skips it (CI checks it with `--package installer-gui`).
- `rootshell/` — small setuid root shell installed on devices.
- `telcom-parser/` — LTE RRC message parser. `src/lte_rrc.rs` is **autogenerated** from the ASN.1 specs in `specs/` via hampi's `rs-asn1c` — never hand-edit it; regenerate per `telcom-parser/README.md`.

Adding a device means touching: `Device` enum in `lib/src/lib.rs`, `daemon/src/display/` and `daemon/src/battery/`, and an installer module. See `doc/porting.md`.

## Common commands

```sh
# Frontend must be built first (daemon embeds it)
cd daemon/web && npm install && npm run build

cargo check
cargo test                       # all default workspace members
cargo test -p rayhunter          # one crate
cargo test -p rayhunter test_name  # single test
cargo fmt --all                  # CI enforces cargo fmt --all --check
cargo clippy                     # CI runs with RUSTFLAGS=-Dwarnings

# Frontend (in daemon/web/)
npm run lint     # prettier --check + eslint
npm run check    # svelte-check
npm run test     # vitest, single run
npm run format   # prettier --write
```

### Cross-compiling for devices (armv7)

Cargo aliases defined in `.cargo/config.toml`:

```sh
cargo build-daemon-firmware-devel   # works with plain Rust toolchain (rust-lld), rustcrypto TLS
cargo build-daemon-firmware         # release profile, pq-tls; needs arm-linux-musleabihf C cross-compiler
cargo build-rootshell-firmware
```

Dev build+install scripts (bash on Mac/Linux/WSL, PowerShell natively on Windows):

```sh
./scripts/build-dev.sh && ./scripts/install-dev.sh orbic   # replace 'orbic' with device
.\scripts\build-dev.ps1 ; .\scripts\install-dev.ps1 orbic  # Windows
```

`./scripts/install-dev.sh util --help` lists device utilities (shells, file transfer, ADB enable).

### Running the daemon locally (no device)

Set `debug_mode = true` in a config to skip DIAG/display/battery/WiFi (see `doc/installing-from-source.md`):

```sh
mkdir -p ./qmdl && printf 'entries = []\n' > ./qmdl/manifest.toml
cargo run -p rayhunter-daemon -- ./config.toml   # config with qmdl_store_path, port, debug_mode = true
```

### Frontend hot reload

```sh
cd daemon/web
API_TARGET=http://192.168.1.1:8080 npm run dev   # proxies API to a running daemon; UI at localhost:5173
```

## Architecture notes

- **Data flow on device**: `/dev/diag` → `lib`'s `diag_device`/`diag` parsing → QMDL recordings stored via `daemon/src/qmdl_store.rs` (manifest.toml + .qmdl files) → analysis (`lib/src/analysis/`) produces NDJSON reports → served over the HTTP API and rendered by the Svelte frontend. QMDL can also be converted to GSMTAP pcap (`lib/src/pcap.rs`).
- **Heuristics** live in `lib/src/analysis/`, one module per detection (e.g. `imsi_requested.rs`, `null_cipher.rs`, `priority_2g_downgrade.rs`), each implementing the `Analyzer` trait from `analyzer.rs` and registered in the `Harness`. Documented in `doc/heuristics.md`.
- **HTTP API**: routes declared in `daemon/src/main.rs` (`get_router()`). OpenAPI docs are generated by `cargo run --bin gen_api --features apidocs`.
- **Installer USB access** goes through EFF's pinned fork of `adb_client` (git dependency): `nusb` transport on Linux, bundled-libusb transport on Windows/macOS — which is why the installer needs a C compiler on those platforms.

## Gotchas

- **Daemon won't compile / `include_dir` errors**: `daemon/web/build` must exist before any `cargo check/build/test` that touches the daemon — build the frontend first. This applies to `gen_api` too; CI fakes it with `touch daemon/web/build/{favicon.png,index.html.gz,...}` when it only needs the API docs.
- **`npm run build` requires `gzip` on PATH**: the build script runs `vite build && gzip -9 ./build/index.html`; the daemon serves the embedded `index.html.gz`.
- **Windows: opaque "Compiler family detection failed" from cc-rs**: Git for Windows' `mingw64\bin` precedes MSYS2 on PATH and ships older copies of gcc's dependency DLLs, so `cc1.exe` dies with `STATUS_ENTRYPOINT_NOT_FOUND`. Hit when building the installer's bundled libusb. Fix: prepend gcc's own directory to PATH (the `.ps1` scripts do this automatically).
- **Windows: `adb push` drops the exec bit**: NTFS has no executable bit, so a daemon pushed from Windows lands as `0666` and boot fails with `/bin/sh: exec: ... Permission denied` in `rayhunter.log`. Fix: `adb shell 'chmod 755 /data/rayhunter/rayhunter-daemon'` after pushing (`make.ps1` does this). Run chmod as the plain adb user — it owns the pushed file, while `rootshell -c 'chmod ...'` can fail with `Operation not permitted`.
- **Restarting the daemon via `rootshell` → init script can panic with bind EACCES**: `TcpListener::bind` hits Android's paranoid networking (socket needs AID_INET) when the daemon is started through a manual `rootshell` → `/etc/init.d/rayhunter_daemon start` chain. A reboot (the `make.sh`/`make.ps1` flow) starts it cleanly via init.
- **Windows: WiFi tools can't be built natively**: `wpa_supplicant`/`wpa_cli`/`iw` need a musl C cross-compiler. Build in WSL or use a release artifact; the installer warns if a device needs them.
- **CI treats warnings as errors** (`RUSTFLAGS=-Dwarnings`), so code that merely warns locally fails CI. It also enforces `cargo fmt --all --check` and clippy.
- **CI is path-filtered**: jobs only run for changed paths (see `files_changed` in `.github/workflows/main.yml`). Put `#build-all` in the commit message to force a full build. Workspace-level files (`Cargo.lock`, `.cargo/`, `scripts/`, `dist/`) always trigger everything.
- **Installer tests on Windows CI run with `--no-default-features`** (`windows_installer_check_and_test` job).
- **`cargo audit` has deliberate ignores** in `.cargo/audit.toml`, each with a rationale — read them before "fixing" an advisory or removing an entry.
- **`.git-blame-ignore-revs` exists** for the bulk `cargo fmt` commit; configure `git blame` with it when doing archaeology.

## Contributing constraints (from CONTRIBUTING.md)

- Run `cargo fmt` and `cargo clippy` before submitting.
- Keep PRs under ~400 lines of non-test code; discuss substantial features in an issue first.
- Disclose AI-generated contributions; descriptions and comments should be written by you.
- Releases: `scripts/set-versions.sh VERSION_NUM` bumps versions; pushing a tag triggers the release workflow.
